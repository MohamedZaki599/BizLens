# BizLens Constitution

## Product Identity

BizLens is an operational intelligence platform for SMBs.

BizLens is NOT:
* an ERP
* an accounting suite
* a bookkeeping platform
* a finance administration system

BizLens transforms financial activity into actionable operational signals.

## Core Principles

### I. Operational Clarity & Simplicity
The platform must prioritize operational clarity and simplicity above all else. Every feature should answer the user's question: "What should I do Monday?"
**Rationale**: SMB owners are overwhelmed by data; they need clarity, not just more charts.

### II. Actionable Signals
Financial activity must be transformed into actionable signals. We do not provide passive data; we provide nudges that imply a next step (e.g., "Profit dropped because of X, review Y").
**Rationale**: Data without action is a distraction.

### III. Fast Onboarding & Low Friction UX
Fast onboarding and low friction UX are essential for SMB adoption. Users should see value within minutes, not hours of configuration.
**Rationale**: High friction leads to abandonment in the SMB market.

### IV. Explainable Signals & Trust
Every signal generated by the platform must be explainable and trustworthy. Avoid "black box" logic; if we highlight a trend, we must show the driver.
**Rationale**: Users only take action when they trust the underlying analysis.

### V. Principled Avoidance
The platform must strictly avoid enterprise complexity, unnecessary workflows, excessive settings, feature bloat, and fake AI functionality.
**Rationale**: Feature bloat is the primary killer of SMB software usability.

## MVP Scope

The MVP includes:
* onboarding
* transaction management
* signal generation
* operational dashboard
* subscriptions
* budgets
* categories
* AI assistant
* Arabic and English support

The MVP does NOT include:
* teams/workspaces
* advanced RBAC
* automation builders
* enterprise reporting
* complex integrations
* multi-tenant enterprise systems

## Architecture & UI Rules

### Architecture Principles
Architecture must remain a modular monolith: maintainable, lightweight, and simple to reason about. Avoid microservices, Kafka, and premature optimization.

### Frontend & Dashboard Rules
* **RTL Support**: Full and proper RTL support for Arabic is mandatory.
* **Consistency**: Maintain consistent spacing, visual hierarchy, and responsive layouts.
* **No Placeholders**: All buttons and interactions must be functional. Placeholder actions are forbidden.
* **Signal Cards**: Must fully contain metadata, avoid overflow, and maintain fixed alignment.

### AI Assistant Rules
The assistant is an operational copilot. It must analyze context, explain anomalies, and summarize trends. It must NEVER hallucinate or provide fake analysis.

### Empty State Rules
Empty states must guide action and explain value. Passive empty states ("No data here") are forbidden.

## Engineering Workflow

Before implementing any feature:
1. Audit current implementation
2. Identify UX violations
3. Validate against MVP scope
4. Validate RTL compatibility
5. Generate implementation plan
6. THEN generate code

Never generate code immediately without analysis.

## Governance

BizLens succeeds when users understand business risks quickly, trust signals, and take action.

**Amendments**: Changes to this constitution require documentation of the rationale and a migration plan for existing features if principles are redefined.
